## 1. 이게 뭔가 — 축 1과의 관계

이 프로젝트에는 이제 **두 개의 축**이 있다.

| | 축 1 (기존) | **축 2 (이 문서)** |
|---|---|---|
| 질문 | "이 **한 회사**의 재무를 깊게" | "**상장사 전체** 중 조건 맞는 회사 찾기" |
| 방향 | 깊게 (depth) | 넓게 (breadth) |
| 데이터 | 재무제표 전체 + 시계열 + 서술 | 핵심 지표 몇 개 × 전체 회사 |
| 저장 | 회사별 Excel 파일 | **크로스-회사 sqlite DB** |
| 출력 | 감사 가능한 시계열 | 조건 필터 → **랭킹 리스트** |
| 엔트리 | `extractor.py` / `build_timeseries.py` | `screener/` |

**같은 레포에 둔 이유:** `dart_client`(corpCode·API·키), `api_to_timeseries`의 `fetch_year`/`ID_MAP`(account_id 매핑), 그리고 헌법의 *결정성 분리* 철학을 그대로 공유한다. 다만 저장계층(DB)·배치수집은 축 2 전용 신규 인프라다.

## 2. 반드시 기억할 사실 — DART에는 주가가 없다 🔴

DART는 **공시·재무제표**다. 주가·시가총액은 없다. 그래서 스크리닝 지표가 둘로 갈린다.

| 지표 | 출처 | 1단계(현재)에서 가능? |
|---|---|---|
| 매출성장률·CAGR, 영업이익률, 순이익률, ROE, ROA, 부채비율, 유동비율 | **DART 재무만으로** | ✅ |
| EPS, 주식수 | DART `stockTotqySttus` | ✅ (아직 미구현) |
| **PER, PBR, 시가총액, 배당수익률, EV/EBITDA** | **주가 필요 → DART 밖** | ❌ → **2단계** |

**범위 결정(사용자):** 1단계는 **재무-only**. PER류는 `pykrx` 같은 외부 주가소스를 붙이는 **2단계**로 미룬다. 그래야 1단계는 현 자산만으로 완결된다.

## 3. 구조 (오늘 만든 것)

```
screener/
├─ common.py     공통 — 경로·API키 로드·sqlite 스키마/연결
├─ universe.py   ① 상장사 모집단 확정 → data/corp_index.json + companies 테이블
├─ collect.py    ② 배치 수집 — company.json(업종)+fnlttSinglAcntAll(재무) → DB
├─ diagnose.py   ③ 한계 진단 — L0~L4를 실측 숫자로 출력
└─ README.md     (이 문서)
screener.db        sqlite (gitignore)  ← 모든 수집 데이터
data/corp_index.json  상장사 인덱스 캐시 (data/는 이미 gitignore)
```

아직 **안 만든 것:** `metrics.py`(비율 계산), `screen.py`(조건 필터+랭킹). → [§6 로드맵](#6-빌드-로드맵).

### 3-1. DB 스키마 (`common.py`)

```sql
companies(corp_code PK, corp_name, stock_code,
          corp_cls,      -- Y유가/K코스닥/N코넥스/E기타 (company.json)
          induty_code,   -- KSIC 업종코드 (64~66=금융·보험)
          is_financial,  -- 0/1 플래그
          updated_at)

financials(corp_code, bsns_year, fs_div,   -- OFS별도/CFS연결
           재무제표, 계정, 값,
           PK(corp_code, bsns_year, fs_div, 재무제표, 계정))   -- long format

collect_log(corp_code, bsns_year, fs_div,
            status,      -- ok/empty/error
            n_accounts, fetched_at,
            PK(corp_code, bsns_year, fs_div))   -- 체크포인트(resume용)
```

### 3-2. 데이터 흐름

```
CORPCODE.xml ──universe──▶ corp_index.json (상장사 3,965)
                              │
                   collect (corp별)
                   ├─ company.json   → companies(cls, induty, is_financial)
                   └─ fnlttSinglAcntAll(연도×OFS/CFS) → financials + collect_log
                              │
                        diagnose ──▶ L0~L4 한계 리포트
                              │
                  (예정) metrics ──▶ 비율 ──▶ screen ──▶ 랭킹 리스트
```

**장애 격리:** `collect_log`에 (corp,year,fs_div) 단위로 status를 남겨, 이미 받은 건 skip하고 중단 지점부터 resume한다(축 1의 Checkpointer 철학 동일).

## 5. 핵심 — 명확히 정의하고 해결해야 할 한계 (가장 중요)

> **파일럿의 반전:** 가장 걱정했던 L2(계정 결측)·L4(데이터 품질)는 **사실상 비-이슈**였다. fnlttSinglAcntAll 응답이 *오기만 하면* 데이터가 깨끗하다(핵심계정 완비, 회계등식 정확). 진짜 일은 **누구를 수집 대상으로 보느냐(L0)**와 **금융업 처리(L1)**다.

| 한계 | 파일럿 실측 | 심각도 | 상태 |
|---|---|---|---|
| **L0** universe 노이즈 | 7/20사(35%) 재무 무응답 | 🔴 최대 | 정의됨, 미해결 |
| **L1** 금융업 왜곡 | 3/13사 (혼재) | 🟡 중 | 정의됨, 방향만 |
| **L2** 핵심계정 결측 | 0건 | 🟢 비-이슈 | 코드 반영됨 |
| **L3** 연결/별도 | 둘다 11·OFS만 2 | 🟢 거의 해결 | 규칙 확정 |
| **L4** BS등식 품질 | 0건 위배 | 🟢 비-이슈 | 게이트만 추가 |

---

### L0 — universe 노이즈 (🔴 가장 크고, 먼저 풀어야 함)

**정의.** `corpCode.xml`에서 "stock_code 있음"은 **"현재 상장 + 재무 가용"이 아니다.** 상장폐지 잔존·외국기업·신규상장 미제출이 대거 섞여 있다.

**증거 (파일럿 7사 무응답):**

| 회사 | corp_cls | induty | 추정 사유 |
|---|---|---|---|
| 삼성디지털이미징 | E | 27 | 삼성전자 합병 (소멸) |
| 아루히 | E | 64919 | 외국기업(일본계), 정기보고서 없음 |
| 한국고덴시 | E | 2612 | 상장폐지 잔존 |
| 에스마크 | E | 13101 | 상장폐지 잔존 |
| 디케이씨 | E | 26211 | 상장폐지 잔존 |
| 럭슬 | E | 303 | 상장폐지 잔존 |
| 채비 | **K** | 28119 | 신규상장 — 2022~24 사업보고서 아직 없음 |

→ **7사 중 6사가 `corp_cls=E`(기타)**. 단 채비처럼 **살아있는 코스닥(K)인데도** 신규상장이라 empty인 케이스가 있어, cls 필터만으로는 부족하다.

**왜 중요한가.** 3,965사 그대로 전체 적재하면 **~35%가 헛콜**이 되어 API 한도를 낭비하고, 스크리닝 결과에 죽은 회사가 섞인다.

**극복안.** universe 정의를 **격상**한다:
- 기존: `stock_code 있음` (3,965)
- 변경: `corp_cls ∈ {Y, K, N}` **그리고** `최근 N년 중 1년 이상 재무 응답 있음` (≈ 2,600 추정)
- corpCode.xml엔 cls가 **없으므로**, `company.json`을 **1회 sweep**(약 3,965콜, ~17분)해서 cls/induty를 캐싱 → 활성 모집단을 한 번 확정.
- 신규상장(채비)은 "재무 응답이 생기면 자동 편입"으로 자연 처리.

**미결정 → [§7]:** 코넥스(N) 포함? sweep을 한 번에 vs 점진적? cls=E를 무조건 배제 vs "최근 재무 있으면 살림"?

---

### L1 — 금융업 비율 왜곡 (🟡 방향만 정함)

**정의.** 은행·보험·증권은 손익 구조가 일반 제조/서비스와 달라(매출=영업수익, 영업이익 정의 상이) 일반 비율(영업이익률·매출성장률 등)이 **무의미하거나 오해를 부른다.**

**증거 (파일럿 금융 플래그 3사):** 노루홀딩스(지주, induty 642), 글로벌텍스프리(택스리펀드, 66x), 한화플러스제3호기업인수목적(SPAC). → `induty_code 64~66` 플래그는 작동하나, **성격이 제각각**이다:
- **진짜 금융(은행/보험/증권):** 일반 비율 왜곡 → 제외 대상
- **지주사(642):** **연결 기준이면** 일반 비율이 유효 → 살릴 수 있음
- **SPAC:** 껍데기 회사 → 대개 L0(재무 빈약)에서 탈락

**극복안.**
- `screen.py --exclude-financial` **기본 ON** (은행·보험·증권 제외)
- 지주사는 **플래그만** 달고 연결기준으로 포함(사용자 선택)
- 장기: 금융업 전용 지표셋(NIM 등)은 **범위 밖** → 1단계는 "제외"가 정답

**미결정 → [§7]:** "금융"의 경계 — 64·65·66 전부? 642 지주는 별도 취급? 부동산리츠(이지스레지던스, induty 68x)는 금융 아님이지만 손익구조가 특이 → 어떻게?

---

### L2 — 핵심계정 결측 (🟢 비-이슈, 이미 처리됨)

**정의.** 매출액·영업이익·당기순이익·자산/부채/자본총계 중 일부가 비면 그 회사는 해당 조건으로 스크리닝 불가.

**증거:** 파일럿 13사 **전부 결측 0건**. account_id(IFRS 표준코드) 기반 매핑이 잘 작동.

**극복안 (이미 반영).** 결측 계정은 **NaN 처리**하고 해당 조건에서 **자동 탈락**시킨다. 임의로 채우지 않는다(헌법 *환각 통제*). 추가 작업 거의 없음.

**주의:** 비-IFRS(K-GAAP) 소형사가 모집단에 늘면 `account_id`가 `-표준계정코드 미사용-`으로 와 매핑이 약해질 수 있다. 전체 적재 후 결측률을 재측정해야 한다(파일럿은 IFRS 회사 위주였을 가능성).

---

### L3 — 연결 vs 별도 (🟢 규칙 확정)

**정의.** 한 회사가 연결(CFS)·별도(OFS)를 둘 다 공시하면 어느 쪽으로 시계열·비율을 잡을지 정해야 한다.

**증거:** 둘다 11사 / OFS만 2사(SPAC·단일회사 — 종속기업 없어 연결 없음, 정상) / CFS만 0 / 없음 0.

**극복안 (확정).** **연결 우선, 없으면 별도**(`diagnose._best_fs` 규칙). 지주·대기업은 연결이 실체에 맞고, 단일회사는 별도가 곧 전체다.

**미결정 → [§7]:** 비율 비교의 일관성 — 어떤 회사는 연결, 어떤 회사는 별도로 섞이면 스크리닝 공정성에 영향? "전부 별도" 모드 옵션이 필요한가?

---

### L4 — BS등식(데이터 품질) (🟢 비-이슈, 게이트만)

**정의.** 파싱/정합 오류로 `자산총계 ≠ 부채총계 + 자본총계`인 회사는 신뢰 불가.

**증거:** 파일럿 **0건 위배**. API 데이터 정합성 우수.

**극복안.** 축 1 `validate.py`의 BS등식(허용오차 100원)을 **수집 후 1회** 적용해 깨진 회사만 배제하는 저비용 게이트. 파일럿상 거의 안 걸리지만 전체 적재의 안전망.

## 6. 빌드 로드맵

```
[완료] universe(인덱스) → collect(파일럿) → diagnose(한계 정의)
   │
   ├─ (A) metrics.py + screen.py 를 파일럿 13사에 먼저 붙여
   │       "조건→랭킹" 출력이 쓸 만한지 싸게 검증           ← 권장 다음 스텝
   │
   ├─ (B) universe.py 에 cls-필터(company.json sweep) 붙여
   │       활성 모집단(~2,600) 락인 (L0 해결)
   │
   ├─ 전체 적재 (하루치, --all + 한도 status 감지 TODO)
   │
   └─ [2단계] pykrx 주가 어댑터 → PER/PBR/시총
```

**권장 순서: A → B → 전체적재 → 2단계.** 출력이 쓸 만한지 13사로 먼저 확인한 뒤 비싼 전체 적재로 가는 게 안전하다.

## 7. 열린 결정 사항 (내가 정해야 할 것)

천천히 생각해볼 것들. 각 결정이 코드 방향을 가른다.

**모집단(L0)**
1. 코넥스(N)도 포함? (유동성 낮고 공시 부실 경향)
2. `company.json` sweep을 3,965사 한 번에? 아니면 cls를 다른 경로로 더 싸게?
3. cls=E(기타)를 무조건 배제 vs "최근 재무 있으면 살림"?

**금융업(L1)**
4. "금융"의 경계: induty 64·65·66 전부 제외? 지주(642)는 살릴까?
5. 부동산리츠·SPAC 같은 특수 손익구조는 별도 카테고리로?

**지표/비율(metrics)**
6. 1단계에 넣을 비율 목록 확정 (성장성·수익성·안정성 중 무엇무엇?)
7. 성장률 계산에 몇 개년? (YoY=2년, CAGR=3+년)
8. 연결/별도 혼재 허용 vs "전부 별도" 일관 모드 옵션?

**수집 범위**
9. 수집 연도 범위 (예: 2020~2024 5개년?)
10. 갱신 주기: 1회성 스냅샷 vs 분기 자동 갱신?

**스크리닝 UX(screen)**
11. 조건 입력 방식: CLI 플래그(`--revenue-growth ">=0.10"`) vs 설정파일?
12. 출력: xlsx 랭킹 vs build_dashboard 재활용한 HTML?

**2단계(주가)**
13. pykrx 도입 시점과 주가-재무 갱신주기 불일치(주가=일, 재무=분기) 동기화 방식.
