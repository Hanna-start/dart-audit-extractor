# 도구 구성 상세 설명

전체 흐름은 **회사명 → 회사코드 → 공시목록 → PDF 다운 → 표 추출 → 시계열 통합** 6단계. 각 단계를 따로 떼서 모듈로 분리하고, 상위에서 `extractor.py`가 조율하는 구조입니다.

```
extractor.py (CLI 오케스트레이션)
    ├─ dart_client.py    (네트워크/API/스크래핑)
    ├─ pdf_to_excel.py   (PDF → DataFrame → Excel)
    └─ consolidate.py    (여러 Excel → 시계열)
```

왜 이렇게 쪼갰는가 — 각 단계가 **실패 양상이 완전히 다르기** 때문입니다. 네트워크 에러, PDF 파싱 에러, 라벨 매칭 에러를 같은 곳에서 다루면 디버깅이 지옥이 됩니다. 단계별로 결과물이 파일(PDF, xlsx)로 떨어져서, 중간에 멈춰도 재시작이 쉽고 어디서 깨졌는지 한눈에 보입니다.

## 1. `dart_client.py` — DART와 대화하는 부분

여기가 이 프로젝트의 거의 모든 위험이 모여 있는 곳입니다. 네 개 함수만 노출합니다.

### `ensure_corpcode(api_key, cache_path)`

DART OpenAPI `https://opendart.fss.or.kr/api/corpCode.xml` 한 번 호출 → zip 응답 → 내부 XML 추출 → 디스크 캐시.

**왜 캐시하나?** 응답 zip이 ~6MB이고 회사 11만 8천 개. 매번 받는 건 과함. modify_date 갱신이 잦은데, 회사 corp_code 자체는 안 변하므로 한 번 받으면 거의 반영구 사용 가능. 일주일에 한 번 정도 새로 받으면 충분합니다.

**왜 OpenAPI인가, dart-fss 같은 라이브러리 안 쓰고?** 추가 의존성 1개 줄임 + 코드가 30줄도 안 됨. 라이브러리 도입은 디버깅 시 블랙박스가 늘어나는 비용이 큽니다. 특히 DART 같은 비표준 API는 라이브러리가 자기 추상화를 끼얹는 경우가 많아 오히려 방해됨.

### `find_companies(keyword, corpcode_xml)`

XML 파싱 후 회사명에 키워드 포함되는 후보 전체 반환. 정렬 우선순위:
1. 정확 일치 (`corp_name == keyword`)
2. 상장사 (stock_code 있는 것)  
3. 짧은 이름 (보통 본체)

```python
def rank(c):
    exact = 0 if c["corp_name"] == keyword else 1
    listed = 0 if c["stock_code"].strip() else 1
    return (exact, listed, len(c["corp_name"]))
```

**왜 이 순서?** 사용자가 "무신사" 쳤을 때 보통 본체 무신사를 원하지 자회사 무신사트레이딩을 원하는 게 아닙니다. 본체는 상장사면 stock_code가 있고 이름이 가장 짧음. "쿠팡" 같이 본체가 비상장(쿠팡주식회사)이고 자회사가 더 많은 경우엔 대화형 선택지로 떨어집니다.

**XML 파서로 `xml.etree.ElementTree` 사용 — 왜 lxml 안 씀?** 표준 라이브러리만으로 충분히 빠름. corpCode.xml 한 번 파싱에 1초도 안 걸립니다. 의존성 안 늘리는 게 이득.

### `list_audit_disclosures(api_key, corp_code, bgn_de, end_de)`

`/api/list.json` 호출. 핵심 파라미터:

```python
"pblntf_ty": "F"   # 외부감사관련
"page_count": 100
```

`pblntf_ty=F`가 핵심. DART 공시 분류:
- A 정기공시, B 주요사항, C 발행공시, D 지분공시, E 기타, **F 외부감사관련**

F 카테고리에는 감사보고서/연결감사보고서 외에도 회계감사인 변경공고, 외부감사인의 감사보고서 등이 섞여 있어서 `report_nm`에 "감사보고서" 포함된 것만 추가 필터:

```python
if "감사보고서" not in nm:
    continue
kind = "연결" if "연결" in nm else "별도"
```

**왜 페이지네이션 처리?** 회사당 10년치 공시면 보통 한 페이지(100건)에 들어오는데, 정정공시 많은 회사는 100건 넘을 수 있어서 `total_page` 보고 루프 돕니다.

**정정공시 처리:** `[기재정정]감사보고서` 같은 보고서가 같은 사업연도에 또 올라옵니다. 별도 `dedupe_latest()` 헬퍼로 같은 (사업연도 추정, kind) 키에서 가장 늦은 접수일자만 남기게 함. **dedupe를 list 함수가 아닌 별도 함수로 분리한 이유**: 정정공시 자체를 보고 싶을 수도 있어서 (예: 원본과 정정본의 차이를 분석). 호출자가 선택하게 둠.

### `download_pdf(rcept_no, dest_path)` — 가장 어려웠던 부분

DART OpenAPI에 **PDF 직접 다운로드 엔드포인트가 없습니다**. 공식적으론 `document.xml`을 주는데 그건 본문 XML이지 첨부 PDF가 아니에요. 실제 PDF는 웹사이트(`dart.fss.or.kr`)의 뷰어 페이지를 통해 받아야 함.

구현 중 발견한 함정 3개:

**함정 ①: dcm_no가 두 종류 있음.** 뷰어 페이지 HTML 안에 dcm_no가 두 군데 등장:
```javascript
// (A) 본문 XML용
node1['dcmNo'] = "9862296"

// (B) PDF 다운로드용
openPdfDownload('20240409000213', '9862295')
```

(A)와 (B)가 1씩 다른 번호. PDF 받으려면 반드시 (B)여야 합니다. 그래서 `_extract_dcm_no()`는 `openPdfDownload(rcept_no, ...)` 패턴에서 `rcept_no가 첫 인자랑 일치하는 쌍의 두 번째 인자`만 채택. 처음엔 단순히 `dcmNo` 텍스트 매칭으로 (A)를 잡아서 빈 응답을 받고 한참 헤맸습니다.

**함정 ②: User-Agent.** `requests.Session()`은 기본 헤더가 `User-Agent: python-requests/2.X`. DART는 이걸 보면 HTTP 200으로 응답하면서 **본문은 빈 바이트열**을 줍니다 (차단인데 4xx/5xx 안 줘서 더 헷갈림). 브라우저 UA로 덮어써야 정상 동작.

여기서 또 한 번 함정: `Session.headers.setdefault('User-Agent', '...')`는 **기본 UA가 이미 세팅되어 있으므로 덮어쓰지 못함**. 그래서 `sess.headers['User-Agent'] = '...'` 직접 할당으로 변경. 코드에 주석으로 박아놨습니다.

**함정 ③: 3단계 페이지 흐름.** 브라우저에서 PDF 다운로드 버튼 누르면 내부적으로:
1. `dsaf001/main.do?rcpNo=X` (뷰어 페이지 — 세션 쿠키 설정)
2. `pdf/download/main.do?rcp_no=X&dcm_no=Y` (다운로드 안내 팝업 페이지 — 추가 쿠키)
3. `pdf/download/pdf.do?rcp_no=X&dcm_no=Y` (실제 PDF, **Referer 헤더 필수**)

처음엔 3번만 바로 호출했더니 빈 응답. 1, 2단계가 쿠키를 만들어줘야 3번이 통과합니다. Referer 헤더도 단순한 anti-hotlinking 체크에 사용됨. 코드 주석에 정확한 흐름을 명시해 뒀습니다.

**왜 BeautifulSoup 안 썼나?** 정규식으로 충분. `openPdfDownload('...', '...')` 패턴이 명확해서 BS의 트리 탐색까지 필요 없음. 의존성 -1.

## 2. `pdf_to_excel.py` — PDF에서 표를 끄집어내는 부분

핵심 라이브러리는 **`pdfplumber`**. 왜 이걸 골랐는가:

| 라이브러리 | 표 추출 | 텍스트 추출 | 한국어 | 평가 |
|---|---|---|---|---|
| pdfplumber | ◎ 셀 단위 추출 | ◎ 위치 정보 보존 | ◎ | 채택 |
| PyPDF2 | ✗ 표 안 됨 | △ 텍스트만 | △ | 표 안 됨 |
| pdfminer.six | △ low-level | ◎ | ◎ | 직접 구현 부담 |
| camelot | ◎ but 설치 까다로움 | - | ◎ | Ghostscript 의존 |
| tabula-py | ○ | - | ○ | JVM 필요 |

`pdfplumber`는 내부적으로 pdfminer.six를 쓰면서 표 추출까지 깔끔하게 wrap한 라이브러리. DART 감사보고서처럼 grid 기반 표가 명확한 경우 정확도 매우 높음. 외부 의존성(Ghostscript, JVM) 없음.

### 핵심 알고리즘: "페이지 첫 줄 보고 섹션 시작 판단"

DART 감사보고서는 표준 양식이 있어서, **재무제표 4종**(재무상태표/포괄손익계산서/자본변동표/현금흐름표)은 각각 새 페이지에서 시작하고, 첫 줄에 "재 무 상 태 표"처럼 큰 글씨 제목이 옵니다. 이 패턴이 있어서:

```python
SECTION_TITLES = {
    "재무상태표": re.compile(r"^(연\s*결\s*)?재\s*무\s*상\s*태\s*표\s*$"),
    "포괄손익계산서": re.compile(r"^(연\s*결\s*)?(포\s*괄\s*)?손\s*익\s*계\s*산\s*서\s*$"),
    ...
}
END_TITLES = re.compile(r"^(주\s*석|...)")
```

`\s*`를 글자 사이에 넣은 이유: PDF에서 큰 글씨 제목은 "재 무 상 태 표"처럼 글자 사이 공백이 들어갑니다. 디자인적으로 자간을 벌린 거라서 추출 시 공백이 그대로 들어옴.

페이지 첫 줄 보고 "재무상태표 시작!" 마킹 → 다음 섹션 마킹 또는 "주석" 만날 때까지 그 섹션 페이지로 묶음 → 그 페이지들의 표를 합쳐서 한 DataFrame.

**왜 이 단순한 휴리스틱이 통하나?** DART가 양식을 통일해놨기 때문. 외부감사인 다른 회사라도 (삼정/삼일/안진/한영 등) 표지 구성은 같음. 이 가정이 깨지는 케이스는 회사가 자체 양식을 쓰는 매우 드문 경우 — 그때는 페이지 인덱스를 수동 지정해야 함.

### 컬럼 정규화 — `_num` 접미사 트릭

추출된 표의 셀은 모두 문자열. `"1,234,567"`, `"(1,234)"`(괄호 음수), `"△500"`(삼각형 음수) 등 다양. 분석할 땐 숫자 필요.

원본을 보존하면서 숫자 컬럼을 추가하는 방식:

```python
def _attach_numeric_columns(df):
    for i, col in enumerate(df.columns):
        if i == 0: continue   # 첫 컬럼은 계정명
        nums = df.iloc[:, i].map(_normalize_number)
        if nums.notna().sum() >= max(2, len(df)//3):
            out[f"{col}_num"] = nums
```

원본 컬럼은 그대로 두고 `당기_num`, `전기_num` 같은 숫자 컬럼만 추가. **왜 원본 보존?** PDF 추출이 가끔 깨질 때(셀 병합 오류, 컬럼 어긋남 등) 사람이 직접 raw 시트 보고 검증해야 하기 때문. `*_num`만 남기고 원본 지우면 디버깅 불가능.

### 메타데이터 추출 — 사업연도 함정

이 부분은 무신사 데이터 검증할 때 발견한 큰 버그입니다. 처음엔:

```python
years = YEAR_RE.findall(head)   # 앞 10페이지의 모든 "YYYY년"
meta["사업연도"] = max(years)
```

→ 감사보고서 본문에 "2024년 4월 9일에 감사보고서 발행"이 적혀 있어서, 결산연도(2023)보다 큰 2024가 max로 잡힘 → **모든 사업연도가 +1년 잘못 매겨짐**. 발견 후 수정:

```python
END_DATE_RE = re.compile(r"(\d{4})\s*년\s*\d{1,2}\s*월\s*\d{1,2}\s*일\s*까지")
end_years = [int(y) for y in END_DATE_RE.findall(head)]
```

표지의 "2023년 12월 31일 까지" 같은 **보고기간 종료일** 패턴만 잡기. 이게 회계학적으로 사업연도의 정의이기도 함. 처음부터 이 패턴 썼으면 됐는데 "YYYY년"만으로 매칭한 단순 정규식이 함정이었음.

회사명 추출도 비슷: `"주식회사 무신사와 그 종속기업"` → 처음엔 `"무신사와"`로 잡혀서 lookahead로 `(?=\s*와\s+그\s+종속|...)` 추가.

## 3. `consolidate.py` — 여러 Excel을 시계열로

PDF별 Excel 13개에서 핵심 계정만 뽑아 (연도 × 계정) 매트릭스로 변환.

### 핵심 계정 정의

```python
KEY_ITEMS = {
    "재무상태표": [
        ("자산총계", r"자산총계$", None),
        ("자본총계", r"자본총계$", None),
        ...
    ],
    "포괄손익계산서": [
        ("매출액", r"매출액$|영업수익$", None),
        ("매출원가", r"매출원가$", "abs"),
        ("법인세비용", r"법인세비용$|^법인세$|법인세등$", "abs"),
        ...
    ],
}
```

**왜 정규식 OR로 묶었나?** 회계기준에 따라 계정명이 다름:
- "매출액" (제조/유통업) vs "영업수익" (서비스업)  
- "법인세비용" (K-IFRS) vs "법인세등" (K-GAAP 일반기업회계기준)
- "당기순이익" vs "당기순손실" vs "당기순이익(손실)"

한 도구가 K-IFRS와 K-GAAP, 흑자년과 적자년을 모두 처리해야 해서 OR로 변형 다 흡수.

**`"abs"` transform:** 매출원가, 판관비, 법인세비용 같은 **비용 항목은 부호가 회사마다 제각각**. 어떤 회사는 양수로(절댓값), 어떤 회사는 음수(괄호 표기)로 적음. 분석할 때 헷갈리니 비용 항목은 절댓값으로 통일.

### `_norm_label` — 라벨 정규화

원본 셀: `"Ⅰ. 매출액(주16)"` → 정규화 후: `"매출액"`. 세 단계:

```python
t = re.sub(r"\s+", "", str(s))          # 공백 제거
t = _PREFIX_RE.sub("", t)                # 로마숫자/번호 prefix 제거 ("Ⅰ.", "1.", ...)
t = re.sub(r"\(주(석)?[\d,\s]+\)", "", t) # "(주16)", "(주석2)", "(주15,16)" 제거
```

마지막 정규식이 까다로웠음: `"(손실)"`은 항목명의 일부니까 살려야 하고, `"(주16)"`만 지워야 함. `\(주(석)?[\d,\s]+\)`로 "주" 다음에 숫자/콤마/공백만 오는 괄호만 매칭하게 함.

이거 못 잡아서 처음엔 2018년 무신사 별도 매출액이 None이었음 (라벨이 `"Ⅰ. 매출액(주16)"` → 정규식 매칭 실패).

### 당기/전기 컬럼 분리

PDF 표는 보통:
```
| 과목 | 당기(2023) | 전기(2022) |
```

또는 일부 회사는:
```
| 과목 | 주석 | 당기 |     | 전기 |     |
|     |      |    값 | 소계 |    값 | 소계 |
```

`_split_period_columns`는 헤더에 "당"/"전" 표시가 있는 컬럼을 anchor로, 그 다음 익명 컬럼(`col_2_num` 등)을 같은 그룹에 귀속:

```python
state = None
for c in num_cols:
    if "당" in c: state = "cur"; cur.append(c)
    elif "전" in c: state = "prev"; prev.append(c)
    else: (cur if state=="cur" else prev).append(c)  # 익명은 직전 그룹
```

이 휴리스틱 덕분에 한 행에 값이 들여쓰기로 여러 컬럼에 흩어진 양식도 처리 가능. 완벽하진 않지만 90%는 잡힙니다.

### 같은 값이 여러 보고서에서 나오는 문제

2024.04.09 감사보고서엔 2023(당기), 2022(전기) 값이 있고, 2023.04.13 감사보고서엔 2022(당기), 2021(전기) 값이 있음. **2022 값이 두 보고서에서 나옴.** K-IFRS 재작성 때문에 값이 미세하게 다를 수도 있고.

처리:
```python
long_df.groupby(...).agg({"값": "last"})
```

파일을 정렬 순서로 읽으면 더 최신 보고서가 뒤에 와서 `last`가 최신 값. 회계기준 재작성된 값이 있으면 그게 우선. 보수적 접근.

## 4. `extractor.py` — 모든 걸 묶는 CLI

`argparse`로 옵션 파싱하고, `run_pipeline()` 함수가 6단계를 순서대로 실행. 새로 짠 건 거의 없고 위 세 모듈을 호출만 하는 얇은 레이어.

신경 쓴 부분 몇 가지:

**진행 표시.** 각 단계를 `[1/6]`, `[2/6]`처럼 번호 매겨 출력. 어디서 멈췄는지 즉시 알 수 있고, 한 회사 처리에 1-2분 걸리니 사용자가 진행 상황 봐야 안심.

**스킵 로직.** 이미 받은 PDF, 이미 변환된 Excel은 건너뜀:
```python
if dest.exists() and dest.stat().st_size > 1024:
    return dest_path   # download_pdf 내부
```
중간에 끊겨도 재실행이 가벼움. 13개 PDF 중 12개 받고 마지막에 실패해도, 재실행하면 12개는 스킵하고 1개만 다시 받음.

**대화형 회사 선택.** 후보 여러 개면 번호 선택:
```python
if len(cands) == 1: return cands[0]
exact = [c for c in cands if c["corp_name"] == keyword]
if len(exact) == 1: return exact[0]   # 정확 일치 1개면 자동
# else 사용자 input 요청
```
무신사처럼 정확 일치 하나만 있으면 자동, "쿠팡"처럼 여러 개면 묻기.

**자동 검증.** 마지막에 회계등식 (자산=부채+자본, 매출=원가+매출총이익) 자동 체크. 이게 어긋나면 추출이 어딘가 깨졌다는 신호. 무신사로 검증 시 별도 9/9, 연결 6/6 ✓ 떠야 안심.

**rate limit.** PDF 다운 사이 `time.sleep(0.3)`. DART OpenAPI는 일 20,000건이라 회사당 13건이면 무시 가능한데, dart.fss.or.kr 웹사이트는 너무 빠르게 때리면 차단 가능성. 보수적으로 300ms.

### stdout 인코딩 함정

처음엔 무신사 폴더 코드에 있던 `sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding="utf-8")`를 그대로 가져왔는데, **PowerShell에서 `py extractor.py` 실행 시 `ValueError: I/O operation on closed file` 에러**. 

원인: pdf_to_excel.py가 import될 때 모듈 최상위에서 stdout을 재포장하는데, PowerShell의 pipe 환경에선 그 시점 stdout.buffer가 잘못된 상태일 수 있음. 한 모듈이 import만으로 stdout 상태를 망가뜨리면 다른 모듈이 print 못 함.

해결:
```python
if __name__ == "__main__":   # 단독 실행 시에만
    sys.stdout.reconfigure(encoding="utf-8")
```
라이브러리 import 시에는 stdout 안 건드림. 호출자(extractor.py)가 인코딩 결정. 그것도 `os.environ.setdefault("PYTHONIOENCODING", "utf-8")`로 환경변수 통해 — 직접 reconfigure하면 같은 closed file 에러 재발하는 환경이 있음.

## 의존성과 선택 이유

| 라이브러리 | 용도 | 대안 검토 |
|---|---|---|
| `requests` | DART API + 웹 스크래핑 | `urllib`(표준)도 가능하지만 Session 쿠키/Referer 관리 코드 길어짐 |
| `pdfplumber` | PDF 표 추출 | PyPDF2(표 안됨), camelot(Ghostscript 필요), tabula(JVM 필요) — 모두 탈락 |
| `pandas` | 표 조작/Excel 출력 | 이거 빼면 너무 손이 많이 감 |
| `openpyxl` | Excel 엔진 | xlsxwriter도 가능. pandas와 호환성·메타 보존 측면에서 openpyxl 우세 |

추가로 표준 라이브러리만 쓴 부분:
- `xml.etree.ElementTree` (lxml 안 씀): corpCode.xml 파싱
- `re`: 정규식 (BeautifulSoup 안 씀)
- `zipfile`, `io`: DART corpCode zip 응답 처리
- `dataclasses`: Disclosure 구조체

**전체 의존성 4개**로 끝낸 게 자랑할 만한 부분입니다. `requirements.txt`가 4줄. 이게 중요한 이유는, 도구를 다른 PC로 옮기거나 6개월 뒤 다시 돌릴 때 환경 깨질 확률이 의존성 수에 비례해서 늘기 때문. 적을수록 안전.
